Slide 2 - The Problem

Title

THE PROBLEM

Lead line

Current agent payment protocols still rely on good faith after payment.

On-slide points

  • Publisher Agent can pay before knowing whether the delivered work will satisfy the brief.
  • Worker Agent can deliver malformed or low-quality output and still get treated like the task succeeded.
  • Neither side has a strong protocol-level proof of what was agreed, what was delivered, and whether it passed.

Bottom line

x402 requests payment. It does not solve verified completion.

Speaker note

"The trust gap is not discovery or pricing. The trust gap is what happens after the money is requested and before the work is accepted."

Slide 3 - How It Works

Title

HOW IT WORKS

Core flow

  1. Publisher Agent requests a job.
  2. AgentLevy returns 402 Payment Required with taskId, serviceId, and committed specHash.
  3. Payment enters escrow on Flare.
  4. Worker Agent submits structured output.
  5. The verifier runs deterministic checks against the committed spec.
  6. Treasury records settlement only if verification passes.

Visuals to place on the slide

  • x402-comparison.svg
  • verifier-process.svg

Hackathon / production line

Hackathon today: deterministic protocol verifier
Production path: attested protocol verifier in confidential compute / TEE-backed infrastructure

Speaker note

"The critical moment is the committed spec hash. Once funds are escrowed, neither side can change the acceptance contract. The verifier checks against that contract, not against a moving target."
